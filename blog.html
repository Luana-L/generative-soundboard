project write-up

generative soundboard

by minseul kim & luana liao

generative soundboard is an in-browser generative music instrument that synthesizes sound live using WebAudio. For a single tune, the user shapes three layers — rhythm, melody, and texture — through controls like Euclidean rhythm settings, pitch class set transformations, FM ratio/index, and many other parameters, and can drive the whole system's tempo with a master BPM slider. Beyond a single tune, three tunes can be combined — played sequentially (Tune 1 → Tune 2 → Tune 3) or stacked together — to mimic the live, iterative workflow of Strudel-style performance while keeping each musical parameter explicit and audible.

1. Inspiration

Our project is inspired by strudel.cc, the in-browser live-coding environment we used in class for our algoraves. Both of us loved how a small change to the code immediately re-shaped the loop you were already hearing. Strudel sits on top of a large pre-built framework, though, so we wanted to see how much of that experience we could re-create using only the raw WebAudio API.

Our professor pointed us at Brian Eno's framing of the generative composer as a gardener: rather than dictating every note, you design the overall environment and let simple rules unfold into music that's never quite the same. We lean on that idea in two specific places — a cellular automaton that drives the melody, and a texture drone that continuously re-harmonizes itself against whatever scale the melody is using. The project also has a teaching motive: we wanted to make Strudel-style parameters (density, transposition, inversion, FM index/ratio, Euclidean steps) explicit enough that you can hear what each one is doing.

2. Constraints

A big part of this project was working under deliberate constraints, since the goal was to mimic strudel.cc using only raw WebAudio. That meant building synthesis, routing, and scheduling ourselves — more flexibility in the kinds of sounds we could make, but also more complexity to manage.

Real-time interaction was the harder requirement. UI changes had to feel immediate without producing clicks or zipper noise, and three independent layers had to stay musically coherent without explicit choreography. To keep parameter edits predictable, every change is staged behind an update music button — the system only adopts new settings when the user commits them. And because JavaScript timers like setInterval can fire unevenly, rhythm scheduled directly from the main thread would feel loose or flam — so we run a lookahead scheduler on the WebAudio timeline instead (see §3).

3. Architecture

Our project is built on three things: one AudioContext, one master gain, and one clock. Every layer is tied to all three.

One context, one master bus

The AudioEngine in src/audio-engine.js owns an AudioContext, plus a master GainNode wired to destination and a small reverb send built from a ConvolverNode with a synthesized impulse response (exponentially-decaying white noise — not a real room, just enough to suggest a room without shipping audio files): 

ensureContext() {
  if (this.ctx) return;
  const Ctx = window.AudioContext || window.webkitAudioContext;
  this.ctx = new Ctx();
  this.master = this.ctx.createGain();
  this.master.gain.value = 0.8;
  this.master.connect(this.ctx.destination);

  this.fxInput = this.ctx.createGain();
  this.reverb = this.ctx.createConvolver();
  this.reverb.buffer = makeReverbIR(this.ctx, 2.4);
  this.reverbOutputGain = this.ctx.createGain();
  this.reverbOutputGain.gain.value = 0.7;
  this.fxInput.connect(this.reverb);
  this.reverb.connect(this.reverbOutputGain);
  this.reverbOutputGain.connect(this.master);
}

We figured out the convolver-based reverb pattern from MDN's ConvolverNode docs and the Web Audio API tutorials: a ConvolverNode applies linear convolution between its input and an impulse response stored as an AudioBuffer. The IR is just an audio buffer — a real-room reverb is one recorded there with a starter pistol, but a synthetic one works too. We fill a stereo AudioBuffer with white noise scaled by an exponential decay envelope to get a nice reverb effect without additional audio files.

A lookahead scheduler

Triggering notes from setTimeout would make rhythm subject to main-thread jitter. To avoid this, we use the lookahead pattern from Chris Wilson's A Tale of Two Clocks — Scheduling Web Audio with Precision: a setInterval wakes us every 25 ms; on each wake we look 100 ms ahead and pre-schedule any events that need to fire before then on the WebAudio clock. The actual oscillator start() calls are sample-accurate; the JS loop only has to keep up loosely. 

scheduler() {
  if (!this.running) return;
  const horizon = this.ctx.currentTime + SCHEDULE_AHEAD_S;
  while (this.nextStepTime < horizon) {
    for (const handler of this.tickHandlers) {
      handler(this.stepIndex, this.nextStepTime);
    }
    this.stepIndex = (this.stepIndex + 1) % 1024;
    this.nextStepTime += this.stepDuration();
  }
}

Per-layer classes

Each layer is a class — RhythmLayer, MelodyLayer, TextureLayer — with the same shape: it owns a GainNode, a reverb send, and a tick handler registered with the engine on attach(). 

attach() {
  const ctx = this.engine.ctx;
  this.gain = ctx.createGain();
  this.gain.gain.value = 0;
  this.gain.connect(this.engine.master);
  this.send = ctx.createGain();
  this.send.gain.value = 0;
  this.gain.connect(this.send);
  this.send.connect(this.engine.fxInput);
  this.engine.onTick((step, time) => this.tick(step, time));
}

Volume and reverb amount are independent AudioParams, so each layer gets its own dry and wet sliders in the UI. Toggling a layer ramps its gain to 0 or its target with setTargetAtTime so we don't get clicks.

The signal flow:

oscillators ─→ layer gain ──┬─→ master ─→ destination └─→ send ─→ convolver ─→ reverb gain ─→ master

4. Layer design

rhythm — Euclidean rhythms + synthesized drums

euclidean bjorklund synthesized drums

Three drum voices — kick, snare, hat — each run their own Euclidean pattern. Euclidean rhythms turn out to underlie a huge chunk of folk and dance music worldwide: the question they answer is, "given n time steps and k pulses, where do the pulses go?" Bjorklund's algorithm answers it by distributing pulses as evenly as possible.

Our implementation is iterative — start with k groups of [1] and n-k groups of [0], then repeatedly merge remainders into the front groups until only one remainder pile is left: 

export function bjorklund(steps, pulses) {
  if (pulses === 0) return new Array(steps).fill(0);
  if (pulses === steps) return new Array(steps).fill(1);

  let groups = [];
  for (let i = 0; i < pulses; i++) groups.push([1]);
  let remainders = [];
  for (let i = 0; i < steps - pulses; i++) remainders.push([0]);

  while (remainders.length > 1) {
    const pairs = Math.min(groups.length, remainders.length);
    const merged = [];
    for (let i = 0; i < pairs; i++) merged.push([...groups[i], ...remainders[i]]);
    const leftover = groups.length > remainders.length
      ? groups.slice(pairs)
      : remainders.slice(pairs);
    groups = merged;
    remainders = leftover;
  }

  return [...groups.flat(), ...remainders.flat()];
}

Each drum voice is synthesized. The kick is a sine oscillator whose pitch sweeps from 120 Hz down to 40 Hz over 180 ms, with a fast attack and exponential amplitude decay: 

function playKick(ctx, dest, time, vel = 1) {
  const osc = ctx.createOscillator();
  const gain = ctx.createGain();
  osc.type = "sine";
  osc.frequency.setValueAtTime(120, time);
  osc.frequency.exponentialRampToValueAtTime(40, time + 0.18);
  gain.gain.setValueAtTime(0.0001, time);
  gain.gain.exponentialRampToValueAtTime(0.9 * vel, time + 0.005);
  gain.gain.exponentialRampToValueAtTime(0.0001, time + 0.4);
  osc.connect(gain).connect(dest);
  osc.start(time);
  osc.stop(time + 0.45);
}

The snare is band-passed white noise plus a triangle thump; the hat is high-passed white noise with a very fast decay. The noise buffers are cached per-context so we're not regenerating them on every hit.

The recipes are standard analog drum-machine moves (Roland TR-808/909). For the underlying principles we leaned on Sound on Sound's Synth Secrets and Andy Farnell's Designing Sound; for the WebAudio mapping we followed MDN's AudioBufferSourceNode docs and Chris Lowis's Synthesising Drum Sounds with the Web Audio API.

The tick callback walks the three patterns and fires any voice whose step is alive: 

tick(step, time) {
  if (!this.enabled) return;
  for (const [voice, track] of Object.entries(this.tracks)) {
    const pos = step % track.pattern.length;
    if (track.pattern[pos]) {
      VOICES[voice](this.engine.ctx, this.gain, time);
    }
  }
}

melody — pitch class sets, walk + CA, FM voice

pitch class sets random walk cellular automaton FM synthesis

The melody layer is where most of the music theory lives. It builds its pitch material from a pitch class set — a list of integers in [0, 11] representing chromatic scale degrees mod octave.

From the set we apply two operations from set theory — transposition (Tn) and inversion (TnI) — both arithmetic mod 12: 

export function transpose(pcset, n) {
  return pcset.map((p) => ((p + n) % 12 + 12) % 12);
}

export function invert(pcset, n) {
  return pcset.map((p) => ((n - p) % 12 + 12) % 12);
}

Then we expand the resulting set across two octaves to get a concrete MIDI pitch pool the melody draws from.

Random walk mode. At each 16th note, with some probability (the density slider), we step one or two scale-degrees from the previous pitch with an occasional larger leap.

Cellular automaton mode. The Eno-influenced part. We keep a 16-cell binary row. Once per bar we evolve it one generation using an elementary CA rule. Each cell looks at its left, center, and right neighbors, packs them into a 3-bit index, and looks up the new value in the rule's bit pattern:

stepCa() {
  const N = this.caState.length;
  const next = new Array(N).fill(0);
  for (let i = 0; i < N; i++) {
    const left   = this.caState[(i - 1 + N) % N];
    const center = this.caState[i];
    const right  = this.caState[(i + 1) % N];
    const idx    = (left << 2) | (center << 1) | right;
    next[i] = (this.caRule >> idx) & 1;
  }
  // nudge out of all-zero / all-one fixed points
  if (next.every((x) => x === 0) || next.every((x) => x === 1)) {
    next[Math.floor(Math.random() * N)] ^= 1;
  }
  this.caState = next;
}

On each beat of the bar, if the cell at that position is alive, fire a note — and the pitch is just the cell index into the pool. You get a constantly mutating melodic line.

Every note is rendered by a small FM synthesis voice: one sine oscillator's frequency modulates another's: 

playFmVoice(carrierHz, time, duration, attack = 0.01) {
  const ctx = this.engine.ctx;

  const carrier = ctx.createOscillator();
  carrier.type = "sine";
  carrier.frequency.value = carrierHz;

  const modulator = ctx.createOscillator();
  modulator.type = "sine";
  modulator.frequency.value = carrierHz * this.fmRatio;

  const modIndex = ctx.createGain();
  modIndex.gain.setValueAtTime(0, time);
  modIndex.gain.linearRampToValueAtTime(this.fmIndex, time + attack);
  modIndex.gain.exponentialRampToValueAtTime(
    Math.max(1, this.fmIndex * 0.2), time + duration
  );

  modulator.connect(modIndex).connect(carrier.frequency);
  // ... amp envelope, start/stop ...
}

texture — detuned saw drone, LFO filter, harmony snapping

detuned saw drone LFO filter harmony snapping

The texture layer is the pad underneath everything. Three sawtooth oscillators, slightly detuned and routed through a low-pass BiquadFilterNode whose cutoff is modulated by a slow LFO. The LFO is itself an OscillatorNode; its output passes through a gain stage that scales it from [-1, 1] into Hz, then connects directly into filter.frequency: 

this.lfo = ctx.createOscillator();
this.lfo.type = "sine";
this.lfo.frequency.value = this.lfoRate;

this.lfoDepth = ctx.createGain();
this.lfoDepth.gain.value = this.lfoDepthHz;

this.lfo.connect(this.lfoDepth).connect(this.filter.frequency);
this.lfo.start();

The Eno-influenced part is harmony snapping. Our initial drone often clashed with the melody, so we made the texture layer listen to the melody's effective pitch class set and re-voice itself whenever that set changes — main wires melody.getEffectivePcset() into texture.setHarmony(pcset) on every update: 

function harmonizedPitches(baseMidi, pcset) {
  const sorted = [...new Set(pcset.map((p) => ((p % 12) + 12) % 12))]
    .sort((a, b) => a - b);
  const baseOct = Math.floor(baseMidi / 12) * 12;

  const allCandidates = [];
  for (let oct = 0; oct <= 3; oct++) {
    for (const pc of sorted) allCandidates.push(baseOct + oct * 12 + pc);
  }

  const root = allCandidates.find((m) => m >= baseMidi) ?? allCandidates[0];
  const snapAbove = (above, targetSemis) => {
    const target = above + targetSemis;
    let best = null, bestDist = Infinity;
    for (const m of allCandidates) {
      if (m <= above) continue;
      const d = Math.abs(m - target);
      if (d < bestDist) { bestDist = d; best = m; }
    }
    return best ?? above + targetSemis;
  };

  return [root, snapAbove(root, 7), snapAbove(root, 16)];
}

We pick the lowest MIDI note in the active set at or above the user's base pitch as the root, then place a second voice roughly a fifth above and a third voice roughly an octave-and-a-third above — both also snapped to set members. The result is a triadic drone that always shares pitches with the melody so that we design the harmonic environment and let parts coexist in them, as Eno describes.

5. UI & performance features

The UI is the instrument. A few choices shape how it feels to play:

  • Staged updates & pending state. Parameter changes don't take effect immediately — they queue up until the user hits update music, which then applies them on a clean boundary. While there are unsubmitted changes, the button glows and the status bar warns the user. This trades a small amount of immediacy for a much more musical feel.
  • 3-page tune sidebar. Up to three full-board snapshots can be saved as separate "tunes." Switch tabs to edit each one independently.
  • Sequence vs. together modes. Once all three tunes are saved, they can be played either in series (Tune 1 → 2 → 3) or stacked simultaneously. This is where the "Strudel workflow" really shows up — sketching variations, then arranging them.
  • CA visualization. When the melody is in cellular-automaton mode, recent generations of the row render below the board, with the currently-firing cell highlighted in pink.

6. Results & future work

What we've achieved

We're proudest of the musical results, with how well the layers fit together and how much customization we got out of a small set of controls. The project is a playground for generative music techniques, but it ended up being fun to play with too, which is exactly what we wanted.

  • Harmony snapping works well. Cycling the melody's Tn or pitch class set on the fly makes the drone glide to a new chord that genuinely fits — there's no awkward beating, because the texture is built from the same pitch material the melody is reading from. It's like the "Eno gardener" moment, where we don't tell the drone what chord to play, but give it the same scale and let it pick.
  • Layered controls compose well. Density, FM index, reverb send, filter cutoff controls let the player achieve very different textures and update easily, while playing continuously. This gave us a lot of control over the music without needing a lot of understanding of the underlying code or music theory, making it even more approachable as a live performance instrument compared to strudel.cc.

What we'd improve next

  • More CA flavors. Right now the rule is fixed, and is only a 1D CA. A 2D CA like the ones shown in class labs would give more interesting patterns to play with.
  • Recording & export. It would super fun to be able to record a session and export it as an audio file, or at least a MIDI file that captures the note events and lets you layer in your own samples or instruments afterward!
  • Visualization. The CA panel is the only layer with a visual. A live waveform/spectrogram on the master would help performers (and viewers) see what they're doing.
